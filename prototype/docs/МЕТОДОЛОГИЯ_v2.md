# Методология оценки нового товара v2 (с MPStats)

Документ фиксирует **итоговую методологию** после обсуждения: что делаем, **почему
именно так**, и какие **подводные камни** у каждого шага. Версия — целевая (под
подключение MPStats). Старые формулы прототипа описаны в `FORMULAS.md`.

Пометка **[ПРОВЕРИТЬ]** = решаем эмпирически в процессе работы (без отдельной
кампании эталонных кейсов — её не делаем).

---

## 0. Что мы решаем (и чего НЕ обещаем)

**Вопрос инструмента:** закупщик фотографирует новый товар + вводит закупочную цену
→ инструмент говорит, **стоит ли брать его на тест в наш розничный магазин**,
опираясь на спрос маркетплейса как на датчик внешнего интереса.

**Чего НЕ обещаем:**
- не обещаем точный прогноз офлайн-продаж (онлайн-спрос ≠ спрос полки);
- для новинки выдаём **размер тестовой партии**, а не годовой объём;
- ВБ — это **датчик спроса**, а не конкурент и не прямой эталон офлайн-продаж.

**Почему так:** между спросом на ВБ и продажами в физическом магазине есть реальный
разрыв. Инструмент честно закрывает часть «есть ли вообще внешний спрос и насколько он
сильный», а финальное решение о закупке всерьёз — после теста в магазинах.
Офлайн-пригодность отдельно НЕ оцениваем и НЕ комментируем.

---

## 1. Решения (что и почему)

| Решение | Почему |
|---|---|
| **Источник данных — MPStats** (заказы, цена, выкуп, детали, похожие, картинки) | реальные данные вместо эвристики отзывы×20; меньше обращений к нестабильному ВБ |
| **Метрика спроса — ЗАКАЗЫ, не выкупы** | заказы = чистый сигнал «товар нужен»; выкупы зашумлены возвратами |
| **Один вызов — дневной график заказов за год** (365 значений) | из него сами считаем 30д, среднемесячное, сезонность, тренд |
| **Заголовок спроса — среднее за 12 мес (год÷12)** | устойчиво и сравнимо между товарами; 30 дней — как «текущая фаза» |
| **Сезонность — всегда рядом** (12 месяцев + индекс) | без неё легко промахнуться по фазе сезона |
| **Спрос: абсолютный порог + перцентиль в категории** (сравнение-с-топом убрано) | для розницы важно «продаётся ли заметно», а не «хит ли на фоне бестселлеров» |
| **Поиск похожих: фото → визуальный поиск ВБ; ссылка/`nm` → метод «похожие» MPStats** | по фото умеет только ВБ. **[ПРОВЕРИТЬ]** что по `nm` точнее — можно оба и сравнить |
| **Картинки похожих — из MPStats** (он хранит ссылку на изображение) | не нужен отдельный поход на basket-CDN ВБ |
| **Похожие товары в выдаче — миниатюра + цена + заказы** | закупщик глазами проверяет, что нашли нужное |
| **Маржа — наценка к рынку ВБ** (`медиана ВБ-цены / закуп`) | базовый шлюз «есть ли запас наценки»; розничную цену закупщик не вводит |
| **УБРАЛИ: концентрацию, снапшоты, чекбокс офлайн-пригодности, розничную цену, дисклеймер про офлайн** | для розницы лишнее / нет честного источника / не нагружаем закупщика |
| **ОТЛОЖИЛИ: вес истории Разноторга** | данные сырые; доработка — следующий этап |
| **Лимиты MPStats — не оптимизируем заранее** | поймём на практике; эталон кэшируем тривиально |
| **Эталонные кейсы (кампанию) — НЕ делаем** | пороги задаём разумным дефолтом и крутим по ощущениям в работе |

## 2. Пайплайн по шагам

### Шаг 1. Вход
**Что:** фото товара (или ссылка/`nm_id`) + **закупочная цена**. Опционально — одно
свободное поле уточнения («500 мл синий L»).

**Почему уточнение одним полем:** ВБ-поиск по фото текст игнорирует; уточнение —
только пост-фильтр по словам в названии. Разносить на поля смысла нет.

### Шаг 2. Поиск похожих — ВСЕГДА визуальный поиск ВБ
- **Фото** → визуальный поиск ВБ (`search-by-photo.wb.ru`, siglip2) → `nm_id` похожих.
- **Ссылка / `nm_id`** → берём картинку товара (из MPStats) → тот же визуальный поиск ВБ.

**Решено (проверено на медведе):** MPStats-`similar` даёт соседей по КАТЕГОРИИ
(мишка → подушка-кирпич, берёзовое бревно) — визуально нерелевантно. ВБ-поиск даёт
именно похожие (мишка → мишки). Поэтому оба пути идут через визуальный поиск ВБ;
MPStats — только данные по найденным артикулам. Если известен сам товар (ссылка/nm),
его спрос показываем ОТДЕЛЬНО («твой товар ~N/мес»), а вердикт ведём по нише похожих.

**Подводные камни:**
- ВБ прячет силу похожести (`cosine = null`) — есть только порядок. Компенсируем
  весом по позиции + совпадением токенов уточнения.
- Визуальная похожесть ≠ тот же товар → защита голосованием категории (шаг 3) и
  показом картинок закупщику (шаг 8).
- Закрытый эндпоинт ВБ может смениться → держим за интерфейсом провайдера.

### Шаг 3. Данные по аналогам + фильтр + категория (всё из MPStats по `nm`)
1. Детали: цена, категория, ссылка на картинку, **дневной график заказов за 12 мес**,
   процент выкупа.
2. **Фильтр наличия:** убираем карточки без остатка/цены (мёртвые листинги).
3. **Фильтр активности:** убираем карточки без заказов.
4. **Категория = взвешенное голосование** (вес = заказы). Лидер ≥ **60%**, иначе UNKNOWN.

**Из графика за год сами считаем:** заказы/30д, **среднемесячное (год÷12)**, 12 месячных
значений (сезонность), сезонный индекс, тренд.

**Подводный камень (доказан на велосипедах):** облако из 256 легко схлопывается до
3–5 живых карточек → код `LOW_SAMPLE`, доверие снижается.

### Шаг 4. Ниша — MPStats `similar` по якорному SKU (стабильно к фото)
Фото/ссылка → визуальный поиск ВБ **только чтобы опознать товар** (якорный SKU) → ниша =
**MPStats `similar` по этому SKU** (до 200 живых карточек, 1 запрос). Почему не визуальные
похожие: их набор **зависит от фото** (обувница: по чистой карточке медиана 489, по фото из
отзыва — 109 → вердикт скакал). similar по SKU стабилен и оценивает **ТИП** товара (для
безбренд-импорта это и нужно — везёшь свой аналог, а не именно эту карточку).

### Шаг 5. Спрос — абсолют в ДЕНЬГАХ к АБСОЛЮТНОМУ полу
```
выкупы/мес   = заказы × выкуп%                                 # реальные продажи (чинит одежду)
выручка_типа = медиана(выкупы аналогов) × медиана(цена)        # ₽/мес типичной карточки типа
абсолют      = выручка_типа ÷ ДЕНЕЖНЫЙ ПОЛ                     # пол = 50 000 ₽/мес (калибруем)
к_топам      = min(1, выручка_типа ÷ медиана выручки топ-10 категории)   # контекст
скорость_s   = 0.65·min(1,абсолют) + 0.35·к_топам
```
**Пол АБСОЛЮТНЫЙ, не относительный к категории.** Раз ниша = similar ≈ категория, то
«выручка_ниши ÷ медиана_категории» ≈ 1 для всего (даже мёртвый степлер с 6к ₽ → GREEN). С
абсолютным полом: степлер→RED, живые типы (150–340к ₽)→брать. Срез категории
(`subject/items`) даёт топ-10 и сезонность; позиция в категории ≈50 (ниша≈категория) —
поэтому в UI показываем «**Выручка типа**», а не позицию.

### Шаг 6. Маржа — мягкий шлюз «запас до рынка»
```
наценка = медианная цена ВБ / закупочная цена
валовая маржа % = 1 − 1/наценка                    (×2 → 50%, ×3 → 67%)
шлюз_маржи: ≥1.3×→1.0 · 1.1–1.3→0.85 · 0.9–1.1→0.65 · <0.9→0.45 (+флаг «рынок ниже закупа»)
```
Не жёсткий RED (офлайн продаёт и дороже ВБ) — маржа только **гасит** балл.

### Шаг 7. Тренд — рост/падение за год
Медиана по аналогам отношения «выкупы за последние 90 дней ÷ за первые 90 дней окна».
`≥1.15 растёт →×1.1 · ≤0.85 падает →×0.85 · иначе ровно →×1.0`.
*Каведат:* окно 365 дней → сравнение не строго сезон-к-сезону, тренд — ориентир.

### Шаг 8. Общий балл (заземлён на GMROII: маржа × скорость)
```
балл_100 = 100 · скорость_s · шлюз_маржи · тренд_коэф
```
Капитал в формулу балла не входит — он уходит в размер теста (шаг 10). «Надёжность»
убрана; выкуп% уже внутри спроса.

### Шаг 9. Вердикт (из балла + отсечки) — ПОЛОСЫ ПРЕДВАРИТЕЛЬНЫЕ
```
жёстко RED:  абсолют < 0.15  → нет спроса
иначе по баллу:
  STRONG — ≥75 и абсолют ≥пола и шлюз ≥0.85   → расширенный тест
  GREEN  — ≥55                                  → тест
  YELLOW — ≥30                                  → малый тест
  RED    — <30                                  (балл держим в красной зоне)
UNKNOWN — нет похожих / нет данных / смешанная категория
LOW_SAMPLE гасит GREEN/STRONG до YELLOW.
```
Полосы **анкерны к «≈ типичный бестселлер категории = GREEN»** и помечены как
**предварительные** (флаг `provisional`) — калибруются на ИСХОДАХ (история Разноторга
или накопленные тесты), не на глаз. Каждая оценка пишется в `saol2/eval_log.jsonl`
(поле `outcome` для факта распродажи) — выборка для калибровки копится сама.

### Шаг 10. Тест-партия — newsvendor (модель газетчика)
```
Cu = медиана_ВБ − закуп          Co = закуп × доля_уценки(деф. 40%)
CF = Cu/(Cu+Co)                  # критический фрактиль
на точку = база(абсолют: ≥1→6 · ≥0.5→4 · ≥0.2→2 · иначе 1) × (0.5 + CF),  RED→0
на сеть  = на точку × число точек (поле необязательное)
```
Дёшево + высокая маржа → CF высок → берём больше; дорого/тонко → меньше. Жёсткий
риск-бюджет НЕ задаём — размер ведут спрос, маржа и закуп.

### Шаг 11. Вывод закупщику
- крупный вердикт + балл + тест-партия **на точку** (и на сеть, если задано число точек);
- спрос в **выкупах/мес** + тренд; рядом заказы и выкуп%;
- наценка к рынку + валовая маржа %; **выручка ниши ₽/мес**; **цена-сегмент объёма**;
- **капитал в тесте** и **потенц. маржа** (если задано число точек);
- сезонность ниши и всей категории (12 столбиков, фаза);
- **похожие товары: топ-5 с миниатюрой + цена + выкупы/мес**;
- коды причин понятным текстом.

### Шаг 12. После теста — реальный GMROII
`GMROII = валовая прибыль ₽ ÷ средний запас по себестоимости` (цель 2–3+) → решение
«раскатывать на сеть / нет». Это закрывает разрыв «спрос ВБ ≠ спрос полки».

## 3. Что осознанно НЕ делаем

- Не оцениваем офлайн-пригодность и не пишем про неё дисклеймеров.
- Не просим розничную цену.
- Не считаем концентрацию/конкуренцию ВБ.
- Не копим снапшоты.
- Не нагружаем закупщика чекбоксами.
- Не используем сырую историю Разноторга в балле (отдельный этап).
- Не собираем кампанию эталонных кейсов — пороги крутим по ощущениям.

---

## 4. Главные подводные камни

1. **Спрос ВБ ≠ спрос полки** — данными ВБ не закрывается; закрывается тестом в магазинах.
2. **Точность ограничена MPStats** — где не знает товар, оценочно с пометкой `SALES_SOURCE_FALLBACK`.
3. **Качество аналогов = качество поиска** — защита: голосование категории, фильтр наличия, картинки закупщику.
4. **Тонкая выборка** — облако часто 3–5 карточек → доверие снижается.
5. **Порог спроса и пороги маржи** — конфигурируемые, версионируемые, покрытые тестами.
